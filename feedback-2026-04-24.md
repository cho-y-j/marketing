# 의뢰자 피드백 정리 & 개선 플랜 (2026-04-24)

> **소스**: 전문 마케터 + 의뢰자 상태 점검 피드백 (2026-04-24 전달)
> **목적**: 11개 피드백 항목을 "현재 문제 → 원인 → 개선 플랜 → 실행 파일" 로 매핑
> **원칙**: 누적 데이터 중심 → **일별 데이터 + 실제 경쟁 기준 + 원인 분석** 중심으로 전환
> **상위 철학**: 자영업자가 "① 왜 매출이 변했는지 ② 지금 뭘 해야 하는지" 를 즉시 이해 가능해야 함

## 0. 전체 요약 — 본질 3줄

1. **경쟁 기준이 "누적 리뷰 많은 순" → "대표 키워드 Top 1~10 매장"으로 이동**해야 함 (의사결정 vs 노출 2-레이어)
2. **모든 지표는 일 단위 delta** (어제/오늘, 전주/이번주) 와 **절대량 + 증가율 이중 표현** 으로
3. **추천은 "분석 결과"로 끝나면 안 됨** — 키워드 → 채널 태그 → 실행 버튼(광고 등록/문구 생성) 까지 연결

## 2. 항목별 상세 (현재 상태 → 문제 → 플랜)

### F1. 홈 화면 — 3중 이슈 [P0]

**피드백**
- 주요 키워드 평균 순위가 표시 안 됨 (확인 필요)
- 매장 등록 시 "매장명 / 주요 키워드" 입력이 정상 안 됨
- 단순 수치 표시 → **"지금 무엇을 해야 하는지"** 구조로 재설계

**확정 방향 (의뢰자)**: 키워드는 **완전 자동 생성** — 수동 입력 필드 없음. 등록 후 편집만 가능.

**현재 상태 분석 필요**
- 홈 페이지: `apps/web/src/app/(dashboard)/(home)/page.tsx` — 평균 순위 위젯 존재 여부 점검
- 매장 등록: `apps/web/src/app/(dashboard)/stores/add/page.tsx` + `packages/api/src/modules/store/store.controller.ts`

**개선 플랜**
1. 매장 등록 폼: **플레이스 URL 1개 입력만** 남김 — 수동 키워드 입력 UI 있으면 제거
2. AI 자동 생성 결과 10개 미만으로 확정, 월 300 미만 제외 (회식 등 예외 유지)
3. 등록 후 `/keywords` 페이지에서 + / × 편집 가능 (이미 존재 → UX 개선 정도)
4. 홈 대시보드 상단을 **"오늘의 액션 3개"** 로 격상 (이미 Phase 2-1 에 존재 — 앞으로 당김)
5. 평균 순위 카드 누락 여부 진단 → 있으면 수치 채우기, 없으면 복원

### F2. 키워드 분석 — "100위 밖" & 일 단위 검색량 [P0]

**피드백**
- "100위 밖"으로 뭉뚱그리지 말고 **실제 순위 (127위, 83위)** 로
- "내 키워드 현황" 영역 의미 불명확
- **일 단위 검색량** 필수 (특히 어제 기준)
  ```
  예: 오창맛집
  어제: 980회  오늘: 720회  ▼ -260
  ```
- 사장님이 "매출 감소 원인"을 데이터로 이해해야 함

**현재 상태 분석**
- 순위 체크: `packages/api/src/modules/keyword/rank-check.service.ts` — **Top 50 까지만 수집** (설정 확인 필요)
- 저장 컬럼: `storeKeyword.currentRank` + `keywordRankHistory`
- 검색량: `storeKeyword.monthlySearchVolume` — 월 단위. 일 단위 저장 구조 없음

**개선 플랜**
1. 순위 수집 범위 **Top 50 → Top 200** 확장 (부하는 병렬 배치로 상쇄)
2. Top 200 밖만 "200위 밖" 표기, 그 외는 실수 순위 표시
3. **일 단위 검색량 스냅샷** 신규 도입:
   - `daily-snapshot.job.ts` 에 `keywordDailyVolume` 모델 추가 — `{keyword, date, dailyVolume, source}` 
   - 네이버 검색광고 API의 `deviceStats.daily` 활용 (현재 월별만 쓰고 있음)
   - 키워드 카드에 "어제 / 오늘 / ▲▼ 증감" 3요소 노출
4. "내 키워드 현황" 라벨 → **"오늘 내 위치"** 로 교체 + 3요소(순위·검색량·유입)로 재구성

**관련 파일**
- `packages/api/src/modules/keyword/rank-check.service.ts`
- `packages/api/src/providers/naver/naver-searchad.provider.ts`
- `packages/api/src/jobs/daily-snapshot.job.ts`
- `apps/web/src/app/(dashboard)/keywords/page.tsx`
- `prisma/schema.prisma` — `KeywordDailyVolume` 모델 추가

### F3. 경쟁 매장 기준 — 대표 키워드 Top 1~10 [P0] ⭐ 핵심

**피드백**
- 현재: 리뷰 많은 순으로 경쟁 매장 표시
- 변경: **특정 키워드 기준 TOP 1~10 매장**을 경쟁 대상으로 설정
- 예: "오창맛집" 검색 시 1~10위 매장

**현재 상태 분석**
- `packages/api/src/providers/data/store-setup.service.ts:130~219` — `pickCompetitorQueries` 가 3~6개 쿼리로 교집합(appearances) 방식으로 선정
- `type: "MAIN"` (대표 키워드) 개념이 DB에 실질적으로 쓰이지 않음 — 모두 `AI_RECOMMENDED`
- 경쟁사는 "공통 노출 매장" 순으로 정렬 → 대표 키워드 1위 매장이 누락될 수 있음

**개선 플랜 — 2-레이어 경쟁사 구조 (의뢰자 A안 확정)**
1. `Competitor.competitionType` 필드 추가: `EXPOSURE` | `DIRECT`
   - **EXPOSURE**: 대표 키워드(`공덕 맛집`) Top 10 — **업종 불문 전부** (카페·일식 포함)
     - 근거: 검색자가 메뉴 미정 상태 → 상권 슬롯 경쟁의 실질적 상대
   - **DIRECT**: 메뉴 키워드(`공덕 아구찜`) Top 5~10 — 같은 업종만
2. AI 프롬프트 (`store-setup.service.ts:841`) 에 `primaryKeyword` 1개 필수 선정 추가
3. 저장 시 `primaryKeyword` → `type: "MAIN"` 플래그
4. `findCompetitorsByQueries` 재설계:
   - Step 1: MAIN 키워드 단독 Top 10 수집 → `EXPOSURE` 저장 (업종 필터 OFF)
   - Step 2: 메뉴 키워드 2~3개 Top → `DIRECT` 저장 (업종 필터 ON)
   - 중복 매장은 두 플래그 모두 on
5. 대시보드 경쟁 비교 화면에 탭 2개:
   - "공덕 상권 강자" (EXPOSURE)
   - "같은 아구찜 경쟁" (DIRECT)

**관련 파일**
- `prisma/schema.prisma` — `Competitor` 모델 수정
- `packages/api/src/providers/data/store-setup.service.ts:130~1252` — 핵심 로직 전면 재작성
- `apps/web/src/app/(dashboard)/competitors/page.tsx` — 탭 구조

### F4. 리뷰 증감률 기준 [P1]

**피드백**
- 급증 기준 1.5% → 5%↑ 로 상향
- 본질: **절대 퍼센트 컷이 매장 규모 차이를 흡수 못 함**
  - 리뷰 100개 매장의 +5개(+5%)와 리뷰 5000개 매장의 +5개(+0.1%)는 체감 다름

**현재 상태 분석**
- `packages/api/src/modules/competitor/` 와 `competitor-refresh.job.ts` 의 임계값 탐색 필요
- 일별 스냅샷 구조는 이미 Phase 8 에서 도입됨

**개선 플랜**
1. 임계값을 단순 % 대신 **이중 컷** 으로:
   - `급증` = (증가율 ≥ 5%) **AND** (절대 증가 ≥ 5개) — 둘 다 만족
   - `증가` = (증가율 ≥ 2%) OR (절대 증가 ≥ 10개)
   - `정상` = 그 외
2. 또는 **Z-score 기반** — 매장 자체 7일 표준편차 대비 몇σ 이탈인지 (더 정확하지만 구현 복잡)
3. 1안으로 먼저 가고, 마케터 재피드백 받아 Z-score 도입 판단

**관련 파일**
- `packages/api/src/modules/competitor/competitor.service.ts` (임계값 상수)
- 경쟁사 알림 판정 로직

### F5. 네이버 실제 순위/매장 수 불일치 [P0] ⭐ 신뢰도

**피드백**
- 네이버 순위 반영과 앱 내 순위 불일치
- 키워드별 노출 매장 수 반영 안 됨
- **데이터 신뢰도 핵심 이슈** (단순 UI X)

**현재 상태 분석**
- `rank-check.service.ts` 의 검색 호출 + 결과 파싱 로직
- 가능한 원인:
  - (a) search.naver.com 가 개인화 결과 반환 (로그인/지역/히스토리)
  - (b) 검색 API 사용 중 → 실제 웹 검색과 랭킹 다름
  - (c) placeId 매칭 불완전 → 같은 매장 중복/누락 카운트
  - (d) HTML 구조 변경 → 정규식 파싱 깨짐

**개선 플랜**
1. **진단 세션 선행** — 현재 "공덕 맛집"으로 직접 검색한 네이버 결과 vs 앱 수집 결과 diff 대조
2. 수집 소스를 단일화 + 버전 고정:
   - search.naver.com **PC 비로그인** UA + 특정 region 쿠키 강제
   - 또는 네이버 Place 공식 검색 API 일원화
3. 매장 수 카운트 로직 재검토 — 유료 광고 슬롯 제외 기준 명시
4. 수집 결과에 **원본 HTML 스냅샷** 보존 (30일) → 추후 diff 가능

**관련 파일**

### F8. 시즌 이벤트 — 채널 태그 & 실행 연결 [P1]

**피드백**
- AI 추천 키워드는 좋음 but "어디에 써야 하는지" 불명확
- 채널 태그 필요: `[플레이스 소식]` `[블로그]` `[파워링크]`
- `+` 버튼 → 단순 추가가 아니라 **"네이버 광고 등록하기"** / **"AI 홍보 문구 생성"** 으로 연결
- 키워드 추천 = 실행 엔진의 입력값이 되어야 함

**개선 플랜**
1. 추천 키워드 카드에 `channels: ["BLOG", "PLACE", "POWERLINK"]` 배열 필드 추가 (AI가 선정)
2. 채널별 배지 색상 코드
3. `+` 버튼 클릭 시 액션 메뉴:
   - "AI 홍보 문구 생성" → 콘텐츠 생성 페이지 딥링크 (키워드 pre-fill)
   - "네이버 파워링크 등록 가이드" → 외부 링크 + 가이드 모달
   - "플레이스 소식 초안 작성" → 콘텐츠 생성 (템플릿 = 소식)
4. 클릭→실행 전환율 수집해 추천 품질 피드백 루프

**관련 파일**
- `apps/web/src/app/(dashboard)/season-events/page.tsx`
- `packages/api/src/modules/content/` — 소식/블로그/파워링크 템플릿 분리
